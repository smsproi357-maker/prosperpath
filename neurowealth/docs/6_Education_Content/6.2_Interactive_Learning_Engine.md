# Feature: Interactive Learning & Quiz Engine

## 1. Purpose
Transforms standard educational content into an active competency gate. It utilizes dynamic simulations, strict multiple-choice decision tests, and algorithmic scoring to objectively verify user understanding before granting completion status.

## 2. Core Functionality
- **Interactive Simulations**: Renders abstract concepts (like order book liquidity) as visual, interactive components requiring user prediction.
- **Decision Tests**: Enforces "mid-module" knowledge checks with immediate binary feedback.
- **Mastery Assessment**: Evaluates comprehensive module understanding via a structured final exam interface.
- **Review Engine**: Generates dynamic breakdown screens mapping user answers against correct values while linking back to contextual definitions.

## 3. Detailed Logic
- **Simulation State (`SIM_ROUNDS`)**:
  - Implements a fixed array object dictating consecutive interaction steps (Round 1, 2, 3).
  - Dynamically builds the DOM representing Order Books (`ob-wrap`) by rendering `ask` arrays and `bid` arrays mapping strictly to variables (`heavy`, `medium`, `thin`).
  - Expects boolean validation (e.g. `STATE._sim_choice === rd.correct`) to trigger positive/negative colored UI feedback.
- **Mastery Assessor (`computeScore()`)**:
  - Validates a multi-step array of user inputs (`STATE.assessment.answers`).
  - Computes `pct = Math.round((correct / total) * 100)`.
  - Enforces a deterministic pass/fail threshold (`passed = pct >= 70`).
- **Review Generator (`buildReviewAnswers()`)**:
  - Iterates over the entire question bank post-submission, injecting custom `.correct` or `.incorrect` CSS classes and appending specific `rv.explanation` context hints for failed strings.

## 4. User Flow
1. User encounters a Simulation screen showing an Order Book scenario.
2. User selects "Up" or "Down" to predict the price action.
3. System immediately provides localized visual feedback indicating correctness.
4. User advances to the Final Assessment containing 5 sequential questions.
5. User selects an answer for Question 1, hitting 'Next'.
6. Upon the final question, user hits 'Submit Assessment'.
7. System calculates score. If <70%, user views 'Review Recommended' layout and retries. If >=70%, competency is 'Verified'.

## 5. Inputs / Outputs
- **Inputs**: Click events on binary choice buttons (`#sim-up`, `#sim-down`), interactions with answer choice strings (`data-ans`).
- **Outputs**: Mathematical percentage scores (`scorePercent`), boolean state updates (`passed`), generated localized CSS color changes (Green/Red success markers).

## 6. Edge Cases
- **Unanswered Submissions**: The final 'Submit Assessment' UI node explicitly intercepts and drops interaction if `hasAnswer` validates as null, freezing the UI securely to prevent null-score bugs.
- **Repetitive Retries**: The engine resets `STATE.assessment` variables explicitly upon hitting the 'Retry Assessment' event handler, natively erasing previous loops without a cache reload.

## 7. Dependencies
- **Course Structure State**: Imbedded explicitly within the surrounding Module State framework variables mapping closely to `STATE.assessment`.
- **Clarify Panel System**: Review answers natively hook into `openClarify()` triggers to provide deep-link definitions for missed questions.

## 8. UI / UX Behavior
- **Feedback Nodes**: Generates specialized `.feedback-box` DOM structures dynamically. Uses pastel-green (`var(--m-green-txt)`) borders for success and desaturated-red borders for logic failures.
- **Progressive Selection**: Answer buttons (`.opt-btn`) visually toggle border-color mapping utilizing `.selected` class overrides while stripping the default hover behaviors of siblings to signify confirmed lock-ins.
- **Terminal Aesthetics**: The order book simulations mirror the actual aesthetic logic of the real `BacktestEngine` order flow maps to ensure visual continuity.

## 9. Future Improvements
- Implement randomized question banks drawing from a larger local JSON repository rather than securely hardcoding the exact 5 questions per file.
- Prevent rapid brute-forcing of simulations by enforcing a 2-second visual delay or cooldown upon incorrect mid-module decision clicks.

## 10. System Role
Serves as the internal validator of the educational layer, ensuring that 'Verified' flags written to the core database represent actual comprehension, rather than just rapid click-through skimming.

# Feature: Portfolio Connection Providers

## 1. Purpose
Manages third-party connection authorizations (Web3 Wallets, Plaid) to synchronize user portfolio states. Centralizes chain metadata, tracks connection sessions, establishes native balance checks, and presents the central visual modal chooser.

## 2. Core Functionality
- **Wallet Orchestration**: Handles native Web3 `window.ethereum` requests targeting MetaMask and Trust Wallet integrations.
- **WalletConnect Bridge**: Provides universal QR code cross-pairing integration utilizing `@walletconnect/ethereum-provider`.
- **Plaid Linking**: Coordinates the native structural link initializations for traditional brokerages via `initPlaidLink()`.
- **Chain Metadata**: Establishes a true isolated Single Source of Truth (`CHAIN_METADATA`) mappings for networks like Polygon, BNB Chain, Optimism.

## 3. Detailed Logic
- **Connection Registry (`PROVIDERS` List)**:
  - Hardcodes the unique `launch()` logic for each provider explicitly.
  - Automatically queries the `window.ethereum` EIP-1102 provider array on runtime to toggle `.available = true | false` depending on active browser extensions.
- **Wallet Provider (`launch()`) Phase Execution**:
  - EIP-6963 provider resolution verifies strict browser injection environments via `isMetaMaskProvider()` / `isTrustWalletProvider()`.
  - Makes specific JSON-RPC data queries over active providers to retrieve native balances (`eth_getBalance`) and network configurations (`eth_chainId`).
  - Registers active provider event callbacks mapping strictly to `accountsChanged` and `chainChanged`.
- **State Store (`ConnectionStateManager`)**:
  - Centralizes current metadata sessions, maintaining active wallets temporarily mapped within `_connectionState.providerData`. Modifies UI dynamically (`_patchWalletUI`) avoiding complex full refreshes.
- **Modal DOM Interactor (`portfolio-connection-chooser.js`)**:
  - Orchestrates a dedicated `pp-connection-chooser` absolute modal.
  - Builds HTML representation iterating cleanly across `crypto` and `traditional` arrays to expose UI buttons.
  - Invokes `window.WalletTokenIngestion.fetchAndRender()` automatically cascading subsequent data fetch phases upon positive socket linkages.

## 4. User Flow
1. User clicks the primary "Connect Account" interaction button on the Portfolio dashboard.
2. Web browser renders the `pp-connection-chooser` native DOM overlay showcasing accessible traditional and crypto integrations.
3. User selects 'WalletConnect' (or standard 'MetaMask').
4. The Chooser logic verifies EVM account bindings, requests address data, stores results, and seamlessly shifts the Global Dashboard UI from 'Disconnected' state variables into 'Connected' overviews mapping wallet contents correctly.

## 5. Inputs / Outputs
- **Inputs**: JSON-RPC EIP connection protocol payloads (`window.ethereum`), User Web3 authorizations, Plaid client bindings.
- **Outputs**: Resolved active address outputs (`user wallet address`), native balance amounts natively transformed by scale division `Number(wei) / 1e18`, updated active specific HTML classes hiding disjointed CTAs.

## 6. Edge Cases
- **Missing External Scripts**: If `walletconnect-provider.js` dynamically fails CDN load, internal error fallbacks (`WCProvider.init` rejection) gracefully alert the native modal instead of corrupting.
- **User Decline Events**: Resolves `4001` signature decline errors seamlessly catching explicit cancels to alert native failure messages neatly rendering in `#pp-chooser-error`.
- **Stale RPC Mapping Vectors**: Directly mitigates failure bindings with generic WalletConnect fallback nodes by explicitly enforcing `.rpcMap` URL assignments natively for all core EVM endpoints inside the script constructor.

## 7. Dependencies
- **EIP Web3 Enablers**: Standard dependencies assuming modern injection.
- **ProsperPath Components**: Highly integrated loosely binding UI element mappings (`#wallet-chain-badge`, `#wallet-address`, `#wallet-summary`, `#disconnected-state`).

## 8. UI / UX Behavior
- **Refined Premium Aesthetic**: Encapsulates dark-mode glassy backdrop filters with highly cohesive visual button behaviors highlighting available vs unconfigured environments naturally.
- **Strict Read-Only Emphasis**: Maintains multiple structural text footprints emphasizing "Read-Only • No Transactions Sent" minimizing potential user security paranoia.

## 9. Future Improvements
- Native integrations expanding specific API links mapping Coinbase / Binance OAuth configurations instead of purely relying upon EVM wallet chains.
- Implementation of multiple concurrent wallet linkages (e.g. hooking 2 distinct Metamask setups consecutively).

## 10. System Role
Operates as the fundamental authentication and network linkage gateway permitting the ProsperPath application to leave closed system boundaries, allowing actual read-only observation capabilities of the users physical financial state.
